# openlist_strm_bridge 工作流程

## —— 系统生命周期、同步及冗余自愈执行流

---

## 一、 启动初始化与自同步生命周期 (核心时序)

程序在启动时，必须严格遵循以下时序，以确保数据库、物理磁盘与云端配置的绝对一致，防止出现时序错乱导致的数据库死锁或文件误删。

```text
    [ 步骤 1 ]  准备环境并初始化数据库 (确保核心表与索引 100% 建立)
        │
    [ 步骤 2 ]  加载引擎配置 (从 Admin API 获取 STRM 存储映射)
        │
    [ 步骤 3 ]  B 区物理磁盘逆向自同步 (盘点物理现状，重建、迁移或自愈)
        │
    [ 步骤 4 ]  同步受保护根目录并扫描已移除根目录
        │
    [ 步骤 5 ]  持久化当前根目录快照
        │
    [ 步骤 6 ]  A 区物理磁盘全量扫描与索引建立 (含 STRM 和字幕文件)
        │
    [ 步骤 7 ]  A -> B 增量同步 (仅同步不存在指纹，打分器过滤劣质命名)
        │
    [ 步骤 8 ]  B 区冗余文件及空目录彻底清理 (保留刮削元数据)
        │
    [ 步骤 9 ]  启动双向 Watchdog 监控与主动刷新定时线程
```

---

## 二、 启动各阶段详细逻辑

## 阶段一：环境准备与数据库自检

- 创建或更新 `a_strm_files`, `b_strm_files`, `strm_identity`, `c_ghost_files`, `subtitles` 等表结构与索引。
- 数据库使用 WAL 模式 (`PRAGMA journal_mode=WAL`)，提升并发性能。

## 阶段二：引擎配置加载

- 登录 OpenList Admin API，获取存储设备列表
- 解析 `addition` 字段，提取 `SaveStrmLocalPath` 和 `paths`
- 构建本地 A 区路径与 STRM 引擎挂载点的映射表

## 阶段三：B区物理磁盘逆向自同步（血统大清洗）

程序在加载引擎配置后，对本地 B区 进行全量底细盘点，确保数据库是物理磁盘的"真实投影"：

1. **收集指纹**：遍历 B区 所有 `.strm` 文件，读取内容并计算指纹。
2. **入库纯新增**：如果在磁盘上发现了指纹，但数据库 B 表中无记录，判定为用户离线拷入的新 strm，血统校验通过后直接写入 B 表并标记为 `valid`。
3. **自愈与自同步（清理失效路径与重命名记录）**：
   - 如果数据库中记录了某条 B区路径，但物理磁盘上既没有这个文件，也没有任何同指纹的"同义路径"（仅改了文件名但内容相同的文件）。
   - 程序将执行 **"B区自同步清理"**：从数据库 B 表和 identity 表中将该失效路径彻底抹除，并在日志中登记。
   - 如果在磁盘上找到了同指纹但不同名字的"同义路径"，程序自动在数据库中执行 `move_b_record`，顺畅完成改名追踪。

## 阶段四：同步受保护根目录

- 从 `strm_engine_paths` 同步受保护根目录到数据库
- 扫描并清理已移除的受保护根目录（迁移到 C 区）

## 阶段五：持久化快照

- 保存当前有效的引擎路径快照，用于后续对比

## 阶段六：A区物理全量扫描

- 遍历 A区 本地目录，解析最新的 STRM 文件并写入 A区 数据库索引
- **同时扫描字幕文件**（`.ass`、`.srt`、`.ssa`），调用 `process_subtitle_file` 进行智能处理：
  - 电影字幕：保持同目录结构，复制到 B 区
  - 番剧字幕：提取季集信息，归档到 `Season XX/` 子目录
- 为接下来的增量同步提供最准确的"事实源"

## 阶段七：A -> B 增量同步

- **指纹防重复制**：遍历 A区 记录。程序只会把**指纹完全不存在于 B区 数据库**的 A区文件复制到 B区。
- **命名保护**：如果该指纹在 B区 已经有了对应的 `valid` 实例（即使物理文件名不同），程序会判定此文件在 B区 已经有更优命名（如用户重命名整理过的名字），**直接跳过复制**，防止劣质原名文件反复回灌、污染媒体库。
- **字幕同步**：STRM 文件复制的同时，同目录下的字幕文件按媒体类型智能归档到 B 区对应位置。

## 阶段八：B区冗余、无效文件与空目录彻底清理

在完成同步后，程序启动最后一轮大扫除：

- **无效/隔离后缀物理清理**：彻底删除本地磁盘上残留的 `.duplicate`, `.quarantined`, `.invalid` 等隔离文件，释放磁盘空间。
- **智能空目录判定**：遍历并清理 B区 的空文件夹。
  - **严苛的安全策略**：文件夹内只要包含任何非 `.strm` 文件，如媒体刮削器产生的 `.nfo` 信息文件、`.jpg`/`.png` 海报图片、`.srt`/`.ass` 外部字幕，**均不算作空文件夹**。程序会予以保留，坚决不破坏用户的刮削元数据！只有完完全全不包含任何物理文件的空目录才会被 `rmdir`。

## 四、 异常与重复文件隔离机制

## 1. 重复文件隔离 (`.duplicate`)

在运行期间，如果 Watchdog 监听到 B区 突然被拷入了重复指纹的文件：

- 评分低（非标准命名）的文件会被 `quarantine_file` 改名为 `xxx.strm.duplicate`。
- 该文件会在下一轮刷新或延迟自愈周期中，被阶段八的清理逻辑彻底物理抹除。

## 2. 损坏与空文件恢复机制

如果 B区 某个 STRM 文件内容被意外修改或置空，导致程序无法解析其 WebDAV 路径：

1. 监听触发。
2. 检索数据库，寻找该物理路径的历史指纹与 A区 源路径。
3. 如果在 A区 找到了原始源文件，立即调用 `shutil.copyfile` 进行**物理恢复**，恢复其 `valid` 状态。
4. 如果 A区 也没有对应源文件，判定为无法挽救的损坏文件，改名为 `xxx.strm.invalid`，标记为 `quarantined`，留待人工排查。

## 3. 单兵越界观察机制

当 B区 某文件被移动到疑似非法位置时：

1. 触发 30 秒延迟观察（`trigger_delayed_solo_check`）
2. 观察期内若有其他同指纹文件跟进，判定为合法批量操作
3. 观察期结束仍只有单文件，且云端有多集 → 物理删除并从 A区恢复
