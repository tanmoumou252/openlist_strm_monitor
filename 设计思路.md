# openlist_strm_bridge 设计思路

## —— 基于 OpenList STRM 引擎同步模式的底层架构设计与边界约定

---

## 一、 项目定位与设计哲学

`openlist_strm_bridge` 是专为 **OpenList STRM 引擎"更新模式"** 打造的本地与云端协调层。它不仅处理 A区（引擎输出目录）到 B区（媒体库扫描目录）的文件同步，更核心的价值在于**将本地媒体库的文件变动转换为对云盘源文件的操作指令**，并在复杂的网络与配置变动中提供多重安全熔断保障。

### 核心设计原则

1. **身份指纹化**：不依赖易变的文件路径或文件名，以 STRM 文件中解析出的规范化云端路径作为唯一指纹键。
2. **血统鉴权化**：B区的任何文件操作必须满足其在 A区的真实血统映射，严防越界与离线剪切导致的文件丢失。
3. **安全自保化**：由于对接云盘，网络不可靠是常态。程序设计以"防灾自保"为第一优先级，在未知状态或网络断连时，选择阻断操作而非盲目清理。
4. **层级结构化**：在云端执行移动或删除时，力求在回收站等虚拟空间内还原真实的本地目录层级。
5. **媒体类型感知化**：自动识别电影与番剧，对字幕等附属文件采取差异化的同步策略，电影保持同目录，番剧按季归档。

---

## 二、 核心技术方案与边界约束

## 1. 引擎配置动态匹配与防误删控制

针对程序监控路径（`refresh_paths.txt`）与 STRM 引擎实际监控范围不一致、或部分网盘突然掉线无法访问的情况，系统设计了动态 API 配置核验方案。

```text
                                [ 探活与比对逻辑 ]
    refresh_paths.txt (程序监控) ────┐
                                     ├─► [交叉分析与探活] ──► 成功 ──► 执行B区冗余清理
    OpenList API 存储白名单 ─────────┘                     └──► 失败 ──► 触发熔断(拒绝清理)
```

- **配置动态提取**：程序启动时从 OpenList API 接口读取所有挂载点，深入解析其 `addition` 字段。兼容 `paths` 字段为 JSON 数组与换行符分隔字符串的两种情况。
- **存储引擎映射**：从 `addition` 中获取 `SaveStrmLocalPath`，与本地 `a_folders.txt` 填写的目录进行精准比对，完成本地物理文件夹与 OpenList STRM 挂载路径（`mount_path`）的强绑定。
- **差异化逻辑决策（Fail-Safe 熔断）**：
  - **程序监控比 STRM 监控少/不相干**：程序仅在交集白名单内执行 B区冗余清理。不在引擎监控下的路径只执行"只读刷新"，绝不执行删减。
  - **云盘掉线/路径不可访问**：探活机制在主动刷新前会对所有引擎路径发起状态查询，一旦节点状态不为 `work` 或请求超时，**断路器立即生效，锁死该节点下的 B区数据**，防止因掉线导致媒体库被全盘抹除。
- **超级极端情况（多引擎重合路径）**：当 `STRM A` 监控 `X, Y, Z`，`STRM B` 监控 `1, 2, X` 时，程序利用 API 返回的对应引擎唯一的 `SaveStrmLocalPath` 建立严格的一对一归属链，确保删除 `B区\大类X` 的文件时，能精准定位到其原本归属的 STRM 引擎挂载点，进而拼凑出正确的云端路径和刷新钩子。

## 2. 虚拟路径与真实云端路径双重映射

程序在执行索引刷新与删除联动时，必须遵循 OpenList 独特的路径映射机制。

- **视频实际云路径**（从 STRM 内容解析）：
  - 格式：`/{真实网盘挂载点}/{大类}/{相对路径}/文件名.mp4`
  - 用途：用于对云端真实文件执行 `move`（移动）或 `remove`（删除）。
- **引擎虚拟入口路径**（用于触发钩子同步）：
  - 规则：OpenList 引擎入口仅显示监控目录的最后一级，中间多余路径会被省略。
  - 映射关系：如果 `mount_path = /strm`，监控的真实路径为 `/天翼云盘家庭云30GB/番剧`。对于真实文件 `/天翼云盘家庭云30GB/番剧/Anime/S01E01.mkv`，其对应的引擎虚拟入口为 `/strm/番剧/Anime/S01E01.strm`。
  - 触发机制：当 B区文件变更或被删除后，必须通过 API 请求 `/api/fs/list` 访问 **虚拟入口目录层级**（例如 `/strm/番剧/Anime`，不带文件名），以此触发 OpenList STRM 引擎的索引更新。

## 3. 云端回收站层级递归重建算法 (`MOVE` 模式)

在 `behavior.action = "MOVE"` 模式下，为了避免被删除的文件在云端回收站内散落一地，程序必须在回收站内完美还原其在本地的目录树结构。

- **回收站路径计算**：
  - 真实路径：`/{网盘挂载点}/{大类}/{层级A}/{层级B}/视频.mp4`
  - 回收站根目录定义：`/{网盘挂载点}/{config.behavior.trash_dir_name}`
  - 映射后的目标路径：`/{网盘挂载点}/{config.behavior.trash_dir_name}/{大类}/{层级A}/{层级B}/视频.mp4`
- **递归重建流程**：
  1. 截取目标回收站目录的所有父级层级（从 `/{网盘挂载点}/{trash_dir_name}` 开始直到最深层级）。
  2. 启动**层级递归探测与创建**：从根层级开始，依次调用 `/api/fs/mkdir` 接口逐层向下创建目录。
  3. 目录树构建完毕后，最终调用 `/api/fs/move`，将源文件移入重建好的层级中，实现完美的回收站结构归档。

## 4. 媒体类型智能识别与字幕差异化处理

程序通过多维度信号自动判断媒体类型（电影 vs 番剧），并据此对字幕文件采取不同的同步策略：

### 4.1 电影模式

- **触发条件**：路径含"电影/movie"等关键词；或目录下仅1个STRM文件且无法提取季集信息
- **字幕处理**：保持与对应STRM文件同目录，命名格式为 `电影名.forced.zho.简体.ass`
- **目录结构**：`B区/电影/xxx/电影名.strm` 与 `电影名.forced.zho.简体.ass` 同级

### 4.2 番剧模式

- **触发条件**：路径含"番剧/anime"等关键词；或STRM/文件名可提取 `S01E01` 季集信息
- **字幕处理**：提取季集信息，归档到 `Season XX/` 子目录，命名格式为 `S01E01.forced.zho.简体.ass`
- **目录结构**：`B区/番剧/xxx/Season 01/S01E01.strm` 与 `S01E01.forced.zho.简体.ass` 同级

### 4.3 语言识别优先级

1. 后缀标识：`.sc`、`.chs`、`.tc`、`.cht`、`.scjp` 等
2. 内容关键词："简中""简体""繁体""中日双语"等
3. 多语种时简中优先标记 `forced`，单语种强制标记 `forced`

## 5. 刮削整理层级边界约定 (刮削适配)

为了不限制用户使用刮削器（如 TinyMediaManager）对 B区文件层级进行重命名与层级划分的自由度，程序对 B区文件的位置变动设置了清晰的合法性边界：

以 STRM 文件指向的**云端真实视频路径的最后一级大类目录**（如 `/番剧/1`）作为校验基准线：

| 变动类型 | 结构示例（基准：`/测试a/番剧/1/02.strm`） | 合法性判定 | 处理逻辑 |
| :--- | :--- | :--- | :--- |
| **加深层级** | `/测试a/番剧/1/s01/ova/02.strm` | **合法** | 属于正常的季别、特辑层级划分，予以保留。 |
| **向上提取** | `/测试a/番剧/1/s01/02.strm` ➡️ `/测试a/番剧/1/02.strm` | **合法** | 从子文件夹向上提回剧集主目录，予以保留。 |
| **同级重命名 (批量)** | `/测试a/番剧/1/02.strm` ➡️ `/测试a/番剧/[2011]1/02.strm` | **条件合法** | 若同级的所有文件一起发生了重命名（或整个剧集本来就只有一集），判定为合法整理，予以保留。 |
| **同级重命名 (单兵)** | `/测试a/番剧/1/02.strm` ➡️ 仅其中一集被移入 `/测试a/番剧/[2011]1/` | **不合法（单兵）** | 判定为用户整理时的手滑或误操作。开启 30秒 观察期，若无后续文件跟进，则物理抹除，并从 A区重新复制原版。 |
| **逃逸至大根目录** | `/测试a/番剧/1/02.strm` ➡️ `/测试a/番剧/02.strm` | **不合法** | 文件脱离了所属剧集文件夹，被提到了大类根目录。判定为非法逃逸，直接物理抹除并从 A区恢复。 |

## 6. 云端服务依赖性与签名时效提醒

由于 OpenList STRM 引擎下发的直链包含具有强校验特性的签名信息（`?sign=`）：

- **令牌强依赖**：该签名与 OpenList 服务端的 Secret / Token 紧密绑定。即便云端路径、文件名称完全一致，只要用户在 OpenList 网页端手动重置了令牌，历史生成的所有 STRM 签名将全部失效，导致媒体库无法播放。
- **系统迁移约束**：这也意味着本地的 STRM 库与当前的 OpenList 服务器是**强绑定**关系。迁移或更换 OpenList 服务器后，必须彻底更新openlist索引(后台>索引>手动扫描>选择strm引擎的入口)，生成正确签名。

---

## 7. 归属判断与跨库污染防御

在多存储或多引擎共存环境下，可能存在同名高评分文件的跨库污染。

- **归属对照原则**：大类 X 文件夹下的所有物理 STRM 文件，其内部解析的云路径指纹必须在 X 对应引擎的白名单内。
- **污染防御**：即便用户将一个命名评分极高、指纹已在数据库中存在的 Y 库文件，手动复制到了 X 库文件夹下，程序在血统校验时，发现该指纹在 X 的 A区对照表中没有对应记录，也会将其判定为"非法外来文件"并予以清除。以此防止媒体库内的文件交叉污染和定位错乱。

---

## 8. 启动时 B 区逆向自同步机制

程序在启动时序中，**加载引擎配置后立即执行** B 区物理磁盘逆向自同步，确保数据库是物理磁盘的"真实投影"：

1. **收集指纹**：遍历 B区 所有 `.strm` 文件，读取内容并计算指纹
2. **入库纯新增**：磁盘上有但数据库无 → 血统校验通过后直接写入 B 表
3. **自愈与自同步**：
   - 数据库有但磁盘无，且无同指纹"同义路径" → 清理失效记录
   - 数据库有但磁盘无，但有同指纹不同名文件 → 自动 `move_b_record` 完成改名追踪
4. **三层校验清理**：幽灵保护 → A区源存在性 → WebDAV存在性

此机制确保：即使程序离线期间用户手动拷入/删除/重命名了 B 区文件，重启后数据库仍能自动对齐物理现状。
